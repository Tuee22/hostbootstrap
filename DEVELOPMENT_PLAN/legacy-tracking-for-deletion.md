> **Purpose**: Authoritative cleanup ledger for the pure-Python surfaces the Haskell-core inversion

## Pending

These surfaces exist in the current pure-Python repository and are slated for removal as the
`hostbootstrap-core` inversion lands. Each entry names its location, why it is slated for removal,
and the owning phase. When cleanup lands, the entry moves to **Completed** in the same change. Per
[development_plan_standards.md § I](development_plan_standards.md).

- **`hostbootstrap/prereqs.py`** — the Python host-prerequisite checks (passwordless sudo, Ubuntu
  24.04, Xcode CLT, Homebrew, Docker socket, NVIDIA runtime, host `ghcup`). The fail-fast host
  minimums shrink to the pre-binary subset and all richer host logic moves into Haskell
  `HostBootstrap.HostPrereqs` plus the `ensure` reconcilers. Owning phase: phase-2 (host trio lift)
  and phase-3 (reconcilers); the residual fail-fast minimums are reclaimed by the thin bootstrapper
  in phase-6.
- **`hostbootstrap/spec.py`** — the dataclass parser that maps rendered `hostbootstrap.dhall` JSON
  into `ProjectSpec` / `ResolvedTarget` / the three model dataclasses and runs residual cross-field
  checks. Superseded by the in-process Haskell skeletal-schema decoder
  (`HostBootstrap.Config.Schema`). Owning phase: phase-4.
- **`hostbootstrap/dhall_tool.py`** — provisions and shells out to a pinned `dhall-to-json` release
  binary to render Dhall to JSON. Superseded by the in-process Haskell `dhall` library decoding the
  skeletal schema directly; no external `dhall-to-json` binary is provisioned. Owning phase: phase-4.
- **`hostbootstrap/models/*`** (`container.py`, `host_binary.py`, `host_daemon.py`,
  `__init__.py`) — the three execution-model implementations the Python CLI dispatches into. The
  three-execution-model abstraction collapses: every project produces one binary through the
  build-twice / copy-out model, and the model split is replaced by substrate-driven build/run logic
  in `hostbootstrap-core`. Owning phase: phase-6.
- **The three-execution-model + `Cluster`/`NoCluster` + `Mount` Dhall schema in
  `hostbootstrap/dhall/package.dhall`** — the rich union schema (`Substrate`, `Model` of
  `Container` / `HostBinary` / `HostDaemon`, `Lifecycle` of `Cluster` / `NoCluster`, per-model
  `Mount` lists, `RModel` / `RLifecycle` / `RTarget` render helpers). Superseded by the skeletal
  schema (`project`, `dockerfile`, `resources {cpu, memory, storage}`) read by Python, with the rich
  project/test Dhall generated by the project binary. Owning phase: phase-4.
- **The `--force-target` model dispatch** in `hostbootstrap/cli.py` — the `--force-target` option
  plus the `isinstance(model, ContainerModel | HostBinaryModel | HostDaemonModel)` branching across
  `_build` / `_run` / `_daemon_run` / `_cluster_up` / `_cluster_down` / `_cluster_delete`, and the
  `_model_name` / `_require_daemon` / `_require_cluster` helpers. The model-keyed dispatch is removed
  with the three-execution-model abstraction; the thin Python bootstrapper has a single
  build-once-and-exec path driven by substrate detection, not a forced model. Owning phase: phase-6.

### Supporting test surfaces (tracked for visibility)

These pure-Python test modules exercise the surfaces above and are removed or rewritten alongside
their subjects, not independently.

- **`tests/test_prereqs.py`, `tests/test_spec.py`, `tests/test_spec_dhall.py`,
  `tests/test_dhall_tool.py`, `tests/test_models.py`** plus the `tests/fixtures/dhall/*` fixtures —
  pinned to `prereqs.py` / `spec.py` / `dhall_tool.py` / `models/*` and the three-model schema.
  Owning phase: tracks the owning phase of each subject above (phase-2, phase-4, phase-6).

## Rules

Per [development_plan_standards.md § I](development_plan_standards.md):

- If an obsolete or duplicate surface still exists, it must appear in the **Pending** section above.
  Each entry names its location, the reason for removal, and the owning phase or sprint.
- When cleanup lands, move the entry from **Pending** to **Completed** in the same change.
- Empty `Pending` and `Completed` sections are valid. The ledger exists as a stable home so cleanup
  obligations are never lost; absence of pending items reflects current reality, not an incomplete
  file.

## Entry format

When a future entry is added, use this shape:

```markdown
- `path/to/obsolete/file` — short reason for removal. Owning phase: phase-N.
```

For more complex entries:

```markdown
- **`path/to/obsolete/surface`** — reason for removal. Owning phase: phase-N, sprint X.Y.
  Replacement: `path/to/new/surface` (see `documents/...`).
```
