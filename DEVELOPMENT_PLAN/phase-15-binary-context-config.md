# Phase 15: Binary Context Config and Command Gating

**Status**: Authoritative source
**Supersedes**: N/A
**Referenced by**: [README.md](README.md), [00-overview.md](00-overview.md), [system-components.md](system-components.md), [binary_context_config](../documents/architecture/binary_context_config.md)

> **Purpose**: Make every project binary explicitly know where it is in the composed
> host/VM/container/cluster chain by reading a sibling `<project>.dhall`, and make normal commands fail
> fast when the local config is missing or does not authorize the command.

## Phase Status

**Status**: Active

**Reopened (2026-06-19)** for the explicit-context / multi-role / forwarded-parameter refinements (see
`## Remaining Work`).

Phases 6, 8, 11, 13, and 14 provide the needed substrate: the thin Python bootstrapper, typed Dhall
generation, the self-reference lift, the worked demo chain, and the composition methodology. This phase
turns the implicit context carried by lift composition into an explicit runtime precondition for every
binary process.

The runtime context is folded into the project-local `<project>.dhall` file. The core normal command tree
gates through the sibling project config, context creation surfaces materialize
host/VM/container/service placement configs, and normal cluster lifecycle commands use the active context
as their runtime authority. The context schema, loader, command gate, topology frames, and runtime
witnesses are built and validated; the Python boundary holds.

This phase was **realigned** to the "chain is the project" model (§ Y, § Z, § X), and the binary-context
contract now matches it: the sibling `<project>.dhall` carries **parameters + context + witness** and never
the chain shape; child-config creation is off the dissolved `context create` mutation verb (the
`context-init` step kind plus `deriveProjectConfigForKind` are the primitives the recursive interpreter
wires inside `project up`, phase-16); the host-level default surface is renamed from `config init` to
`project init` (Python migrated); the read-only `context` command (`renderComposition` + `context inspect`)
absorbs the former `config schema` / `config show FILE` / `config path` / static `config render` surfaces;
and the per-frame fail-fast mechanism (`validateRuntimeContext` over the topology witnesses) is built and
unit-tested. The schema/loader/gate substrate (Sprints 15.1, 15.2, 15.6) and the realigned command-surface
sprints (15.3, 15.4, 15.5) are `Done`; the effectful wiring of the `context-init` step and the per-frame
gate into the recursive `project up` apply is owned by phase-16.

## Remaining Work

Refine the binary-context contract to the explicit-context / multi-role / forwarded-parameter model
(development_plan_standards § X):

- Each `<project>.dhall` carries an **explicit context** and may declare **more than one role** — a single
  config can be both a project (deployment) authority and a `service` authority — and each command checks
  the capability it needs (a service-only config runs `service run`, refuses `project up`).
- A context's relationship to the others is expressed in the **pure compositional lifts** (the
  `topologyFrames` / `parentChain` frame graph), not implicitly.
- Child configs are **generated by the binary from passed parameters, some forwarded from the parent
  context's `<project>.dhall`** — a child config is a parameterized projection of its parent (the
  `context-init` step inside `project up`, and a ConfigMap overriding the baked container config for a
  service pod, § AA).
- The read-only `context` command treats **all** `<project>.dhall`s uniformly, regardless of the roles a
  config declares.
- Forward dependency: the `project up` interpreter that wires `context-init` and the per-frame gate is owned
  by [phase-16](phase-16-project-lifecycle-command.md); the service-role config is consumed by
  [phase-18](phase-18-service-runtime-command.md).

The realigned contract built earlier remains valid and validated:

- `context` is a **read-only composition-introspection** command (`HostBootstrap.Context.renderComposition`
  + `context inspect`, `ContextSpec`), absorbing the former `config schema` / `config show FILE` /
  `config path` / static `config render` surfaces; no mutation.
- The standalone `context create vm|container|service` verb is removed; the `context-init` step kind
  (`HostBootstrap.Step`) plus `deriveProjectConfigForKind` (`SchemaSpec`) are the primitives the recursive
  interpreter wires inside `project up` (the effectful step and its real-run validation owned by phase-16).
- The host-level default-config surface is renamed `config init` -> `project init` (§ Y); the Python
  bootstrapper triggers `project init --if-missing` (139 Python tests green).
- The **per-frame fail-fast mechanism** is `validateRuntimeContext` over the declared topology witnesses,
  built and unit-tested; the recursive-handoff gating of `project up` is owned by phase-16.
- `.dhall` = **parameters + context + witness**, never the chain shape; structural variation is a
  root-`<project>.dhall` flag, so the chain stays a pure function of root parameters (§ Y).

The `project` lifecycle command, the `context-init` chain step, and the recursive interpreter that carries
the per-frame fail-fast handoff are **new work owned by phase-16** (`phase-16-project-lifecycle-command.md`,
Project lifecycle command and step-chain interpreter); this phase reopens the binary-context contract those
changes rest on.

## Phase Objective

Introduce the runtime binary-context contract:

- Every normal binary command reads `<project>.dhall` from next to the executable before dispatch.
- The project name in the filename is derived from the Cabal file/binary identity; the role is inside the
  file content.
- The built project binary creates defaults through `config init` and emits schema/help for editing.
- Each nested boundary receives or creates its own role-specific local config before the nested binary
  runs.
- The Dockerfile path uses
  `<project> config init --role image-build-container --output /usr/local/bin/<project>.dhall` after
  installing the binary and before `check-code`; runtime containers receive parent-generated configs for
  actual topology frames.
- Kubernetes service pods receive a service/daemon local config from their owning controller; durable
  services use a `StatefulSet`.
- Missing, undecodable, wrong-project, wrong-capability, or wrong-command contexts fail fast with exit
  code 1.
- Missing or unverifiable topology witnesses also fail fast before side effects.

#### Remaining Work

#### Objective

Use the project-local `<project>.dhall` file as the single runtime config authority for each binary copy.

#### Deliverables

- Sibling-file discovery looks for `<project>.dhall`, where `<project>` is derived from the running

- Daemon/service startup logs include project, binary, context kind, role name, config path, config hash,

- Config changes during a process lifetime do not mutate the active process; normal commands read once,

#### Remaining Work

Sibling `<project>.dhall` discovery, the runtime-context section, the command/capability gate, the
read-once semantics, and the daemon/service startup logging are built and stay valid. The **config-surface
names** that materialize and read that authority change under the new model and are the target being
built:

- The host-level default surface (`config init --role host-orchestrator ...`) is renamed to `project init`
  with the same optional `--cpu` / `--memory` / `--storage` / `--ha-replicas` parameters (§ Y); Python

  `config init --role image-build-container --output /usr/local/bin/<project>.dhall` to the equivalent

The `project init` / `project up --dry-run` surface that re-expresses these renames is **new work owned by

### Sprint 15.6: Topology frames and runtime witnesses [Done]

**Status**: Done
**Implementation**: `core/hostbootstrap-core/src/HostBootstrap/Context.hs`, `core/hostbootstrap-core/src/HostBootstrap/Config/Schema.hs`, `demo/docker/Dockerfile`, `demo/src/HostBootstrapDemo/Commands.hs`

#### Objective

Make the Dhall context topology-aware so each binary has a complete picture of the composition chain and
can fail fast when the process is not running in the declared frame.

#### Deliverables

- Add an execution topology to `<project>.dhall`: frames, parent references, provider kind, current frame,
  capabilities, command classes, resource envelope, and runtime witnesses.
- Keep the frame graph open-ended so higher layers can represent arbitrary chains such as host -> Lima VM
  -> Docker container -> kind cluster -> pod or host -> VM -> Pulumi role -> EKS cluster -> workload.
- Add local witness checks for implemented providers: host, Lima VM, Incus VM, Docker runtime container,
  and Kubernetes service contexts.
- Change Dockerfile defaults from VM-project-container authority to image-build authority.
- Require parent-generated runtime configs for lifted `test all`.

passes; `cabal run hostbootstrap-demo -- deploy --dry-run` renders the VM-local
