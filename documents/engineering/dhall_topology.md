# Dhall Topology

**Status**: Authoritative source
**Supersedes**: N/A
**Referenced by**: [documents-index](../README.md), [schema](schema.md), [resource_budgeting](resource_budgeting.md), [hostbootstrap_core_library](../architecture/hostbootstrap_core_library.md), [python_haskell_boundary](../architecture/python_haskell_boundary.md), [binary_context_config](../architecture/binary_context_config.md)

> **Purpose**: Define the Dhall configuration topology: the Python-read static bootstrap input, the
> per-binary context config, and the binary-generated rich/test tiers.

## TL;DR

- Configuration is typed Dhall with a strict bootstrap/runtime split.
- The static-base `hostbootstrap.dhall` is read only by the Python bootstrapper and is identical in
  shape across projects.
- Every project binary reads a sibling `project-binary-context-config.dhall` before normal command
  dispatch. This runtime context tells the binary where it is in the global composition chain.
- The rich project-level Dhall and the per-case test-harness Dhall are generated by the project binary.
- `hostbootstrap-core` owns the reusable static/bootstrap and context substrate; rich/test schemas are
  artifacts the project binary emits.

## The Configuration Set

| Config | Shape | Produced by | Read by |
|------|-------|-------------|---------|
| Static bootstrap `hostbootstrap.dhall` | `project`, `dockerfile`, `resources {cpu,memory,storage}` | Hand-authored per project; identical shape across projects | The thin Python bootstrapper only |
| Binary context `project-binary-context-config.dhall` | Project identity, context kind, parent chain, capabilities, allowed command classes, resource envelope | Created during bootstrap and at each nested boundary | The project binary before normal command dispatch |
| Rich project-level Dhall | Runtime roles plus cluster-bootstrap instructions | Generated by the project binary | The project binary |
| Per-case test-harness Dhall | One typed value per test case | Generated by the project binary | The project binary / test harness |

## Static Bootstrap — `hostbootstrap.dhall`

The static-base file is the only Dhall the Python bootstrapper reads, and it is identical in shape
across every project: `project`, `dockerfile`, and the `resources` budget. Its full shape is documented
in [schema](schema.md); the budget field is detailed in [resource_budgeting](resource_budgeting.md).

Normal project-binary execution does not read this file. Python reads it pre-binary, builds the
host-native executable, and writes the first sibling `project-binary-context-config.dhall`. The
in-process `HostBootstrap.Config.Schema` decoder remains for bootstrap support and the explicit
`config show FILE` inspection path.

## Binary Context — `project-binary-context-config.dhall`

Every project binary reads a context file next to itself before dispatching any normal command. The file
is created by the bootstrap path or by a parent context before the binary is launched in a nested context.
It tells the binary whether it is the host orchestrator, the VM binary, the VM-container test runner, a
cluster service, a daemon, or another project-specific role. Commands whose semantics do not match the
declared context fail fast with exit code 1.

The context tier is described in
[binary_context_config](../architecture/binary_context_config.md).

## Rich And Test Dhall

The rich project-level Dhall (runtime roles plus cluster-bootstrap instructions) and the per-case
test-harness Dhall are *artifacts the project binary emits*. The project binary also emits its own
schema: the rich schema is generated, not owned by `hostbootstrap-core`.

- **WRONG**: add the project's roles/cluster fields to `hostbootstrap.dhall`.
  This is wrong because it pushes project-specific shape into the Python-read bootstrap input, breaks
  the "identical static-base shape across projects" rule, and forces the thin Python layer to understand
  runtime semantics it must not.
- **RIGHT**: the project binary generates its rich Dhall and emits its own schema; normal binary
  commands read the sibling binary-context config first, then read any richer generated config that is
  valid for that context.

This split keeps a single canonical home per concern: the static-base contract is owned by
the Python bootstrap path, the binary-context contract is owned by the project-binary runtime, and the
rich runtime/test contracts are owned and generated by each project binary.
