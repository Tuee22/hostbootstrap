> **Purpose**: Define the target Dhall configuration topology: binary-local project config, generated
> child configs, and binary-generated rich/test artifacts.

## TL;DR

- The old Python-owned `hostbootstrap.dhall` input is removed. Python derives the project name from the
  Cabal file and does not read or write Dhall.
- The canonical local runtime config is the executable's sibling `<project>.dhall`.
- Every normal command reads that sibling file before dispatch and fails fast when it is missing or does
  not authorize the command.
- Parent binaries generate narrower child `<project>.dhall` files at VM, container, daemon, and service
  boundaries. Children never reach back to read the parent's host config.
- Rich deploy/project Dhall and per-case test Dhall remain binary-generated artifacts with schemas emitted
  by the project binary.

## Current Status

Python no longer participates in Dhall. The project binary owns default local config generation, pure child
projection helpers, and normal command gating through the sibling `<project>.dhall`. The runtime context
authority lives inside that local config alongside project-owned settings.

## The Configuration Set

| Config | Shape | Produced by | Read by |
|------|-------|-------------|---------|
| Local runtime `<project>.dhall` | Project settings plus runtime context authority | Generated by `<project> config init`; edited by the user for host-level settings; generated by parents for child contexts | The project binary before normal command dispatch |
| Generated child `<project>.dhall` | Narrow projection of host/project settings plus child authority | Parent binary during `cluster up`, VM bootstrap, Dockerfile build, or service/daemon materialization | The child binary copy |
| Rich project/deploy Dhall | Runtime/deploy records composed from the reusable vocabulary | Generated by the project binary | The project binary |
| Per-case test-harness Dhall | One typed value per test case | Generated by the project binary/test harness | The project binary / test harness |

## Local Runtime Config

`<project>.dhall` lives next to the executable that reads it. It carries two concerns in one local file:

- user/project settings, such as Dockerfile path, resource budget, deploy knobs, HA replicas, ports, and
  feature flags;
- runtime context authority, such as context kind, role name, parent chain, capabilities, allowed command
  classes, resource envelope, and child-context rules.

The role is data inside the file, not a filename suffix. This keeps lookup deterministic while still
letting host, VM, ad-hoc container, daemon, and service copies use different local contents.

## Generated Child Configs

When a command crosses a boundary, the parent binary first generates the next local config at the target
path. The child config is a projection:

- it carries only the settings the child needs;
- it carries a narrower context and allowed command set;
- it carries the appropriate resource envelope or budget slice;
- it never grants unrelated host, Docker-build, VM-orchestration, or service authority.

For example, a host config may contain the full project budget and deploy settings. A cluster-service
config should contain only the service role, its local resource envelope, and the service settings it needs
to serve.

## Rich And Test Dhall

The rich project/deploy Dhall and per-case test-harness Dhall are artifacts the project binary emits. The
project binary also emits their schemas. `hostbootstrap-core` owns the reusable vocabulary and the context
spine; project-specific rich schemas are generated by the project binary.

- **WRONG**: make a service pod read the host's `<project>.dhall` directly so it can see fields such as
  replica count or image tag. This is wrong because it leaks host-level authority and couples a child to a
  parent file path.
- **RIGHT**: the parent reads its own config, validates it, and generates a narrower service
  `<project>.dhall` before the service starts.

This split keeps a single canonical home per concern: Python owns only pre-binary build mechanics, the
local `<project>.dhall` owns the binary's current role and settings, and generated rich/test artifacts are
owned by the project binary.
