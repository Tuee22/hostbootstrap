# Dhall Topology

**Status**: Authoritative source
**Supersedes**: N/A
**Referenced by**: [documents-index](../README.md), [schema](schema.md), [resource_budgeting](resource_budgeting.md), [hostbootstrap_core_library](../architecture/hostbootstrap_core_library.md), [python_haskell_boundary](../architecture/python_haskell_boundary.md)

> **Purpose**: Define the three-tier Dhall configuration model and the rule that the rich schema is
> a project-binary artifact while `hostbootstrap-core` owns only the static-base-schema decoder.

## TL;DR

- Configuration is typed Dhall in three tiers.
- The static-base `hostbootstrap.dhall` is read by the Python bootstrapper and is identical in shape
  across projects.
- The rich project-level Dhall and the per-case test-harness Dhall are generated by the project
  binary.
- `hostbootstrap-core` owns only the static-base-schema decoder; the rich and test schemas are
  artifacts the project binary emits.

## The Three Tiers

| Tier | Shape | Produced by | Read by |
|------|-------|-------------|---------|
| Static-Base `hostbootstrap.dhall` | `project`, `dockerfile`, `resources {cpu,memory,storage}` | Hand-authored per project; identical shape across projects | The thin Python bootstrapper |
| Rich project-level Dhall | Runtime roles plus cluster-bootstrap instructions | Generated by the project binary | The project binary |
| Per-case test-harness Dhall | One typed value per test case | Generated by the project binary | The project binary / test harness |

## Tier 1 — Static-Base `hostbootstrap.dhall`

The static-base file is the only Dhall the Python bootstrapper reads, and it is identical in shape
across every project: `project`, `dockerfile`, and the `resources` budget. Its full shape and decoder
are documented in [schema](schema.md); the budget field is detailed in
[resource_budgeting](resource_budgeting.md).

`hostbootstrap-core` owns the decoder for this tier and *only* this tier (`HostBootstrap.Config.Schema`).
Keeping the Python-read schema small and uniform is what lets the bootstrapper stay thin — see
[python_haskell_boundary](../architecture/python_haskell_boundary.md).

## Tiers 2 and 3 — Binary-Generated Dhall

The rich project-level Dhall (runtime roles plus cluster-bootstrap instructions) and the per-case
test-harness Dhall are *artifacts the project binary emits*. The project binary also emits its own
schema: the rich schema is generated, not owned by `hostbootstrap-core`.

- **WRONG**: add the project's roles/cluster fields to the static-base schema in `hostbootstrap-core`.
  This is wrong because it pushes project-specific shape into the shared decoder, breaks the
  "identical static-base shape across projects" rule, and forces the thin Python layer to understand
  project semantics it must not.
- **RIGHT**: the project binary generates its rich Dhall and emits its own schema; the binary reads
  those tiers itself, while `hostbootstrap-core` continues to decode only the static-base tier.

This split keeps a single canonical home per concern: the static-base contract is owned by
`hostbootstrap-core` and consumed by Python, while the rich runtime and test contracts are owned and
generated by each project binary.
